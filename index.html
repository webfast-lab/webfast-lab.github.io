从 IPv4 / IPv6 获取 MAC 的实现总结

2026-03-25
Linux 系统编程>网络编程
词
  • IPv4
  • IPv6
  • Linux
  • MAC
  • Netlink
  • 网络编程

问题背景

在 Linux 环境里，IP -> MAC 不是一个单一接口就能直接覆盖所有情况。

如果目标 IP 就是本机某个网卡地址，那么应该返回本机接口的 MAC。
如果目标 IP 是同链路邻居，那么应该优先从内核邻居缓存中获取。
如果目标 IP 不在本地链路上，那么真正能拿到的通常不是目标主机的 MAC，而是下一跳网关的 MAC。

/home/ryk/Documents/code/c/ip_to_mac.cpp 这份代码，正是按这个顺序做了一个统一查询入口。

整体流程

程序入口是 ResolveMacAddress，逻辑分成三步：

  1. 先识别传入的是 IPv4 还是 IPv6。
  2. 先查本机接口地址是否命中。
  3. 未命中时查邻居表；再失败则回退到网关路由并查询网关的邻居项。

这个顺序比较合理，因为它符合 Linux 网络栈里的真实转发路径。

关键实现拆解

1. 先判断地址族

代码通过 DetectFamily 调用 inet_pton，自动识别输入是 IPv4 还是 IPv6，并把二进制结果写入 in_addr 或 in6_addr。

这样后续所有逻辑都可以共享一套接口，只在底层按 AF_INET 和 AF_INET6 分支处理。

2. 本机接口命中时直接返回

FindLocalInterfaceMac 使用 getifaddrs 遍历本机接口：

  • 遇到 AF_PACKET 地址时，记录接口名和对应 MAC。
  • 遇到 AF_INET 或 AF_INET6 时，判断是否与目标 IP 完全相等。

如果目标 IP 正好是本机某张网卡绑定的地址，就直接返回这张网卡的 MAC。

这一层避免了一个常见误区：本机地址并不需要通过 ARP 或 NDP 去“解析”。

3. 同链路目标从邻居表拿 MAC

FindNeighborMac 通过 netlink 向内核发送 RTM_GETNEIGH 请求，读取邻居缓存。

它在 ParseNeighborResult 里重点解析两个属性：

  • NDA_DST：邻居项对应的目标 IP
  • NDA_LLADDR：链路层地址，也就是 MAC

只要发现 NDA_DST 与目标地址匹配，并且同时存在 NDA_LLADDR，就能返回结果。

这部分同时兼容 IPv4 和 IPv6，本质上统一走的是 Linux 邻居子系统：

  • IPv4 对应 ARP 邻居缓存
  • IPv6 对应 NDP 邻居缓存

4. 邻居表未命中时回退到网关

如果目标不是同链路邻居，程序会转去 FindGatewayMac。

它先调用 FindGatewayRoute 发起 RTM_GETROUTE 查询，拿到目标 IP 对应的路由结果，重点提取：

  • RTA_GATEWAY：网关地址
  • RTA_OIF：出接口索引
  • RTA_PRIORITY：路由优先级

拿到网关 IP 后，再次调用邻居查询逻辑，去找这个网关在邻居缓存中的 MAC。

这一点很关键：对于跨网段目标，二层真正发送帧时使用的是“下一跳”的 MAC，而不是远端主机本身的 MAC。

为什么要同时支持 IPv4 和 IPv6

这份实现里把很多基础能力都做成了“按地址族复用”的方式：

  • DetectFamily 负责统一识别输入
  • FamilyAddressSize 负责统一计算地址长度
  • SockaddrEqualsIp 负责统一比较接口地址
  • FindNeighborMac 和 FindGatewayRoute 同时支持两种协议族

这样代码结构更稳定，不会出现 IPv4 一套逻辑、IPv6 再单独复制一套的问题。

代码里值得保留的设计点

RAII 管理系统资源

代码里定义了 NetlinkSocket 和 IfaddrsGuard 两个小封装：

  • NetlinkSocket 负责自动关闭 netlink fd
  • IfaddrsGuard 负责自动释放 getifaddrs 返回的链表

这种写法能明显减少异常路径或早返回时的资源泄漏风险。

结果对象统一

最终返回值统一为 MacLookupResult，里面包含：

  • macAddress
  • interfaceName
  • source

这样主流程不需要关心底层是从本机接口、邻居缓存还是网关路径得到的结果，接口比较干净。

实际限制

这类程序并不能保证“任何 IP 都查得到 MAC”，原因主要有三类：

  • 目标不在本地链路上，只能得到网关 MAC。
  • 内核邻居缓存里没有对应项，且程序本身没有主动触发 ARP/NDP 探测。
  • 输入 IP 非法，或者系统当前没有可用路由。

源码在失败路径里也把这些情况写得比较明确，这一点对排障很有帮助。

总结

如果把这份程序概括成一句话，就是：

先判断目标是不是自己；不是自己就查邻居；邻居没有再按路由找下一跳。

这套思路贴近 Linux 实际的二层转发与邻居解析过程，也因此比“直接查 ARP 表”更完整，尤其适合同时处理 IPv4 和 IPv6 的场景。

模运算基本性质的推导

性质描述

对于整数 ( x ) 和 ( y )，若 ( x % k = y % k )，则 ( x - y ) 是 ( k ) 的整数倍（即 ( x - y = mk )，其中 ( m ) 为整数）。

推导过程

已知条件

给定：
[ (A + B) % k = (B + C) % k ]

应用模运算性质

根据上述性质，两数同余则差为 ( k ) 的倍数：
[ (A + B) - (B + C) = mk (m为整数) ]

化简表达式

化简左边的式子：
[ (A + B) - (B + C) = A - C ]

因此得到：
[ A - C = mk (即 A = C + mk) ]

证明 ( A % k = C % k )

模运算的定义：对于任意整数 ( x )，( x % k ) 表示 ( x ) 除以 ( k ) 的余数，记为 ( r )，满足：
[ 0 < r < k ]
且：
[ x = nk + r (n为整数) ]

证明步骤：

  1. 设 ( C % k = r )（即 ( C = nk + r )，( 0 leq r < k )）
  2. 代入 ( A = C + mk ) 得：
    [ A = (nk + r) + mk = (n + m)k + r ]
  3. 根据模运算定义，( A % k ) 即为 ( A ) 除以 ( k ) 的余数 ( r )，因此：
    [ A % k = r = C % k ]

题解思路

综上所述，我们证明了：
当[ (A + B) % k = (B + C) % k ]成立时， A % k = C % k也成立。

因此能连接的序列，其B数字旁边的其余两个数字的取模必然是相同的，则易发现最终得到的序列取模必然是a, b, a, b, a, b这种结构的

因此若当前k为3的时候，取模得到的值可能为0, 1, 2

则可能出现的序列即为[0, 0], [0, 1], [0, 2], [1, 0], [1, 1], [1, 2], [2, 0], [2, 1], [2, 2]

归并排序

算法原理

归并算法是对分治思想的实现。结合到算法中，一个无序的有理数序列，把它两两分成多个子序列，对这些子序列进行排序，此时多个子序列是有序的，然后再两两子序列结合，再次进行排序，持续进行这个过程，当前序列逐渐变得有序。这就是基本的算法思路。

编写思路

递归思路：既然当前是需要将无序序列分成多个子序列，那么定义出来一个左边界和一个右边界，然后在递归当前左边界和有边界，直到当前序列中只剩一个元素，则当前序列就是有序的，然后在对两两序列进行排序，逐渐向上排序，最后获得一个有序的序列。

快速排序

算法原理

通过取到一个标志位上的值，区分出小于这个值的部分和大于这个值的部分，再对两边进行排序。本质上来说，就是找出当前排序节点在这个序列中的位置。逐步的将当前大问题分为多个小问题去处理。

编写思路

既然上面已经明确了，当前我需要找出一个标准的点（pos值）去将当前的序列进行划分，那么不妨取当前序列的最后一个值（或中间值或左值均可）。当我对这一个值进行排序之后，那么这个值对于序列来说，这个index对应的值只可能是它或者相同值的数。那么，接下来需要面临的，此时我需要明确如何去递归的实现，即自上到下，从大的序列到小的序列逐渐缩小范围，最终得到的序列即为有效序列。

现在解决了当前的递归问题，剩下的就是如何去让当前一个序列分成大于pos值和小于pos值的两个区间。
那么我取最后一个值为pos值，现在让我的index停留在left-1的位置。此时，如果当前值大于pos的值，那么index就不能变动，因为index的值是后续pos需要交换的值。反之，小于pos时，此时交换当前值交换到index的位置。即如图所示。
